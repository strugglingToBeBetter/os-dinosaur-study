# 1. 서론
## 1.1. OS 기본
1. OS 개요
- OS(Operating System: 운영체제)란 컴퓨터의 HW를 제어하는 소프트웨어다.
- OS는 컴퓨터(HW)와 사용자 간의 상호작용을 연결하는 매개 시스템 역할을 한다.

2. OS의 역할
- OS는 시스템 자원을 사용자나 앱에 할당하는 역할을 한다.
- OS는 시스템 커널에 접근해서 시스템 자원(메모리, 디스크, CPU 등)을 관리한다.
- 모바일 디바이스에서는 OS는 미들웨어를 포함하기도 한다(iOS, Android)
- OS는 사용자가 시스템을 편리하게 사용할 수 있도록 도와주는 역할도 한다.
- 사용자 입장에서는 컴퓨터/OS와 다양한 방식으로 상호작용한다(마우스/키보드 입력, 터치스크린, 음성인식 등).
- 임베디드 디바이스의 경우 사용자의 개입 없어도 프로그램이 작동할 수 있도록 디자인된 경우가 많다.

3. OS 지식의 필요성
- 모든 응용 프로그램들은 OS를 기반으로 생성/동작하므로, OS에 대해 알아야 프로그램을 적절하고, 효율적이고, 효과적이고, 보안적으로 취약하지 않게 만들 수 있다.
- 또한 프로그램을 변경하거나(write) 사용할 때에도 유용하다.

## 1.2. 컴퓨터-시스템 구성
1. 개요
- 현대의 일반적인 컴퓨터 시스템은 CPU와 디바이스 컨트롤러가 버스(bus) 형태로 연결되어 구성 요소(component)와 공유 메모리의 접근을 허용하는 방식으로 구성되어 있다.
- 각 디바이스 컨트롤러는 특정 타입의 디바이스를 담당한다.(예: 디스크 드라이브, 오디오 장치, 그래픽 표시 장치 등)
- 컨트롤러에 따라서 여러 개의 디바이스가 연결될 수 있다(예: USB 포트로 여러 대의 디바이스 연결하기).
- 운영체제는 '디바이스 드라이버'를 통해 디바이스 컨트롤러를 식별하고 디바이스가 시스템의 잔여 메모리에 접근할 수 있도록 한다.

2. 인터럽트
- 인터럽트는 시스템이 처리 중인 작업을 일시적으로 중단하게 하는 기능이다.
- 디바이스(컨트롤러)가 드라이버(디바이스 드라이버)에게 디바이스의 상태(작업 완료, 처리 중 등)를 알려주는 용도로 사용한다.
- CPU가 인터럽트 신호를 받으면 즉시 고정된 위치(*인터럽트 프로세서 주소를 뜻하는 것 같다)로 작업을 전환한다.
- 인터럽트에 해당하는 작업이 완료되면, CPU는 인터럽트 신호를 받기 전에 수행하던 작업을 재개한다.
- 인터럽트는 수시로 발생하므로, 빠르게 취급(handle)되어야 한다.
- 인터럽트 아키텍처는 인터럽트에 대한 상태 정보를 저장하고 있어야 한다.

3. 인터럽트의 실행
- 인터럽트의 기본 매커니즘은...
  ```
  1. CPU가 인터럽트-요청 라인(interrupt-request line)에서 컨트롤러의 신호를 감지한다.
  2. CPU는 인터럽트 번호를 읽고 인터럽트 핸들러 루틴으로 건너뛴다(jump).
  3. 인덱스와 연관되어 있는 메모리 주소에서 인터럽트 작업을 시작한다.
  4. 인터럽트 핸들러는 (인터럽트 동작 중) 변경되는 상태를 저장하고, 인터럽트 발생 원인을 설정하고, 인터럽트 이전 상태로 돌아가기 위한 필수 프로세스 실행을 담당한다.
  5. 인터럽트 핸들러가 'return_from_interrupt'를 실행하여, CPU를 인터럽트 이전에 다른 작업을 실행하던 상태로 되돌린다.
  ```
  의 방식으로 진행된다.
- 인터럽트의 기본 매커니즘은 CPU가 비동기적인 이벤트에 대응할 수 있도록 설계되었지만, 현대의 OS는 인터럽트 처리 기능이 더 복잡해졌다.
- 현대의 인터럽트는...
```
1. 주요 작업 중에는 인터럽트 동작 처리를 뒤로 미루거나
2. 디바이스에 적절한 인터럽트를 보내기 위한 적절한 방법이 필요하거나
3. OS가 인터럽트 간 우선순위를 판단하여 반응 순서를 조절할 수 있는 다중 레벨(multilevel) 인터럽트가 필요하다.
```

4. 인터럽트 작동 예시 및 요약
1) CPU가 디바이스의 입출력 초기화를 명령
2) 디바이스 입출력 초기화
3) 입력 준비 완료, 출력 완료, 오류 발생 등의 인터럽트 신호를 생성하여 CPU에 전달
4) CPU는 인터럽트 신호를 수신하여, 인터럽트 핸들러에 통제 권한 이전
5) 인터럽트 핸들러가 데이터를 처리하고, CPU를 인터럽트 이전 상태로 되돌림
6) CPU는 인터럽트 이전에 수행하던 작업을 재개함

5. 인터럽트 요청 라인
- CPU는 nonmaskable interrupt와 maskable interrupt 라인을 갖고 있다.
- maskable interrupt 라인은 방해받으면 안 되는 중요한 동작이 있는 경우 꺼질(turned off) 수 있다.
- maskable interrupt는 디바이스 컨트롤러가 서비스를 요청할 때 사용된다.

6. 다중 인터럽트 처리
- CPU는 다양한 디바이스(와 이에 비례하는 인터럽트 핸들러)를 보유하고 있으므로, 인터럽트 핸들러를 매칭하는 시간/자원을 최적화할 필요가 있다.
+인터럽트 벡터 주소보다 컴퓨터가 제공 가능한 인터럽트 처리기가 더 많으므로, 고유 값을 할당할 수 있는 방법이 필요한 것도 있음
- 인터럽트 체인(chaining): 인터럽트 벡터 안의 각 요소가 인터럽트 헨들러 리스트의 헤드를 가리킨다. 인터럽트 발생 시 요청을 처리할 수 있는 인터럽트가 발견될 때까지 리스트의 핸들러가 하나씩 호출된다.
- 인터럽트 체인은 인터럽트 우선 순위 수준을 판단할 때도 사용한다.
- 인터럽트는 반응에 민감한 프로세스에서 중요하게(heavily) 사용되므로, 효율적인 인터럽트 처리는 시스템 퍼포먼스 향상에 필수적이다.

7. 저장소 구성
- CPU는 메모리에만 명령을 적재(load)할 수 있으므로, 모든 프로그램은 실행 전에 반드시 메모리에 적재되어야 한다.
- 프로그램이 실행되기 전 적재되는 메모리는 메인 메모리(RAM)이라고 한다.
- RAM은 휘발성이 있어서 컴퓨터 전원이 꺼지면 저장된 정보를 상실한다.
- RAM의 단점을 보완하기 위해 컴퓨터는 전자적으로만 삭제 가능한 형태로 프로그램된 읽기 전용 메모리(EEPROM)와 기타 형태의 펌웨어를 사용한다.
  *이 때 사용하는 '기타 펌웨어'는 비휘발적이고 자주 write되지 않은 저장소이다.
- EEPROM은 자주 변경될 수 없고, 쓰기 속도도 느리다. 그래서 자주 변경되지 않는 정보(고유 정보 등)를 저장할 때 사용하기도 한다.

8. 메모리 구성
- 모든 형태의 메모리는 바이트 배열을 제공한다.
- 각 바이트에는 고유한 주소가 있다,
- 적재(load)는 메인 메모리의 바이트나 단어를 CPU 내 레지스터에 옮기는 작업이다.
- 저장(save)은 레지스터의 내용물을 메인 메모리로 옮기는 작업이다.
- CPU는 프로그램 카운터에 저장된 위치부터 실행하기 위해 메인 메모리에서 자동으로 명령을 적재한다.

9. 폰 노이만 사이클(vonNeumann architecture)
- 폰 노이만 사이클에서는...
```
1. 명령을 메모리에서 페치(fetch)하고, 페치한 명령을 명령 레지스터에 저장(store)한다.
2. 명령이 해독되면 메모리에서 피연산자가 페치되어 내부 레지스터에 저장될 수 있다.
3. 피연산자에 대한 명령이 실행되면, 실행 결과는 메모리에 다시 저장될 수 있다.
```
- 우리는 실행 중인 프로그램이 생성하는 메모리 주소의 시퀀스에만 관심이 있으므로, 메모리 주소가 생성되는 과정은 무시할 수 있다.
- 이상적으로 우리는 프로그램 데이터가 메인 메모리에 영구적으로 존재해 있기를 원하지만, 메인 메모리는 용량이 작고 휘발적이어서 현실적으로 불가능하다.
- 그러므로, 모든 컴퓨터 시스템은 메인 메모리의 확장 역할로서 보조 기억 장치(secondary storage)를 제공한다.

10. 보조 기억 장치
- 보조 기억 장치는 많은 양의 데이터를 영구적으로 저장할 수 있어야 한다.
- 주로 쓰이는 보조 기억 장치는 HDD(*요즘은 대신 SSD를 많이 씀), 비휘발성 메모리(NonVolatile Memory) 장치가 있다.
- 보조 기억 장치는 메인 메모리보다 처리 속도가 느리므로, 적절한 관리는 컴퓨터 시스템에서 가장 중요하다.

11. 기타 저장장치 구조(삼차 저장소)
- 넓은 의미에서 (레지스터, 메인 메모리, 보조 기억 장치로 구성된) 저장소 구조는 수많은 가능한 저장소 시스템 디자인 중의 일부에 불과하다.
- 캐시 메모리, CD ROM(*요즘으로 치면 USB 메모리), 마그네틱 테이프 등도 저장 장치의 일종이다. 이것들을 삼차 저장소(tertiary storage)로 부른다.
- 각 저장소 시스템은 데이터를 오랫동안 보관할 수 있는(나중에 데이터를 불러와도 검색할 수 있도록) 기능을 제공한다.
- 삼차 저장소 시스템은 구성이 다양하지만, 주요 차이점은 처리 속도, 크기, 휘발성 여부이다.
- 일반적으로 속도와 크기는 반비례하는 관계이며, 작고 빠른 메모리는 CPU에 가깝다.

12. 저장 장치 계층
- 저장 장치 계층은 레지스터-캐시-메인 메모리-NV 메모리-HDD-광학 디스크-마그네틱 테이프 순서로 형성되어 있다.(*뒤로 갈 수록 저장 공간이 커지는 대신 처리 속도가 느려짐)
- 상위 4개 계층(레지스터, 캐시, 메인 메모리, NV 메모리)은 반도체로 구성되어 있다.
- NVM의 경우 일반적으로 하드 디스크보다 접근 속도가 빠르다. NVM은 주로 플래시 메모리를 사용하는데, 모바일/임베디드 영역에서 많이 쓰인다. 최근에는 컴퓨터나 서버의 장기 데이터 저장에도 사용한다.
*기억 장치 구성은 경제적으로 최적화된 형태여야 한다(*용도와 필요에 맞게 사용). 두 구성요소(휘발성-비휘발성 저장소) 간 성능 차이가 큰 경우 캐시를 설치해 이를 보완할 수 있다.

## 1.3. 입출력 구조
- 운영체제 코드의 주요 부분은 입출력(I/O) 제어에 할애되어 있다.
- 입출력 제어는 시스템의 신뢰성과 성능에 주요한 영향을 주고, 장치의 속성이 다양하기 때문에 제대로 관리하는 것이 중요하다.
- 인터럽트 구동 I/O 형태는 소량의 데이터 제어에는 유용하지만, NVS I/O처럼 대량의 데이터를 이동할 때는 시스템에 높은 오버헤드를 유발할 수 있다.
*오버헤드: 어떤 작업을 처리하기 위해 드는 자원(시간, 메모리, 제어 신호 등)
*NVS: 비휘발성 저장소(Non-Volatile Storage)
- 위 문제를 해결하기 위해, 직접 메모리 접근(Direct Memory Access: DAM)을 사용한다.
- 장치에 대한 버퍼 및 포인터, 입출력 카운트를 세팅한 후, 장치 제어기는 CPU의 개입 없이 메모리로부터 자신의 버퍼 장치 또는 버퍼로부터 메모리로 데이터 블록 전체를 전송한다.
- 블록 당 하나의 인터럽트만 생성되며, 디바이스 드라이버에게 '동작 완료'함을 알린다.
- 디바이스 컨트롤러가 작업들을 수행하는 동안, CPU는 다른 작업을 할 수 있다.
